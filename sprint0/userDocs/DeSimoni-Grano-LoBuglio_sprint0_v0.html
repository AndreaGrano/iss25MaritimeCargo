    1. Introduction

    A Maritime Cargo shipping company intends to automate the operations of load of freight in the ship’s cargo hold. To this end, the company plans to employ a Differential Drive Robot for the loading of goods in the ship's hold.

    1.1 Our motto

    There is no code without a project, there is no project without problem analysis, there is no problem without requirements.

    1.2 Aims of sprint0

    The aims of the current sprint are:

    • Formalization of the requirements analysis given by the customer, in order to make them machine-understandable
    • Agreement on the meanings of the requirements from both parties
    • Drafting of a general model of the system
    • Drafting of a primitive interaction schema
    • Design of executable test plans to show the customer the main behaviour of the system

    1.3 Requirements

    A primary interaction with the customer pointed out the following requirements:

    • The products must be registered in a database by specifying its weight using a proper service (productservice)
    • At the end of the registration process, productservice returns a positive integer Unique Product Identifier PID
    • The hold is a rectangular, flat area with an Input/Output port (ioport)
    • There are 4 slots provided for storage
    • There is an additional slot, but it's permanently occupied, so it acts like an obstacle (slot5)
    • When the product is registered, a load request can be sent. The outcome is postive if the weight is under a certain threshold (MAX_WEIGHT) and if there is still at least 1 slot available
    • If and when the load request is accepted, cargoservice associates a slot to the product PID and returns the name of the reserved slot
    • After the association, cargoservice waits for the product to be delivered at the ioport. In the meantime, other requests will not be elaborated
    • cargoservice is able to detect (by means of the sonar sensor) the presence of the product container at the ioport when a distance less than DFREE/2 is measured for a reasonable time (e.g. 3s)
    • cargoservice is able to ensure that the product container is placed by the cargorobot within its reserved slot
    • After placing the load in the reserved spot, cargorobot should return to its HOME location, then cargoservice can start processing another load-request
    • cargoservice is able to show the current state of the hold, by mesans of a dynamically updated web-gui.
    • cargoservice interrupts any activity and turns on a LED if the sonar sensor measures a distance D > DFREE for at least 3s. The service continues its activities as soon as the sonar measures a distance D ≤ DFREE

    2.3 General overview

    2.3.1 Software provided by the customer

    • BasicRobot - software component which executes movement commands of a DDR-robot, regardless of the technology with which the robot is implemented (virtual or real)
    • RaspberryPi utility software
    • SonarLed2025 - a software already implemented and usable which allows to get the distance from the sonar and to blink a LED

    2.3.2 Bounded contexts

    From the analysis of text provided by the customer we found out that the software system is going to comprehend 5 boundend contexts at least:
    • Loading management/Cargo process - this context is a mastermind that interacts with other contexts in order to manage requests by checking the weight, assigning product's PID to a slot if available, detecting if there is a container at the IO port and delivers it to the assigned slot;
    • Hold inventory / slot management - this context manages slots knowing which ones are available or occupied. It also knows the products' PIDs associated with occupied slots;
    • Physical IO / robot/sensor gateway - this context interacts with sonar detecting if a container is available at the IO port;
    • Hold visualization / reporting - this context provides data and manages the GUI in order to show the state of the hold dinamically;
    • Product management - this context manages products' lifecycle and it provides registration service, it associates a given product with a PID and stores product's details

    3. Requirements formalization

    3.1 Entities

    Entity Activity Type Description/Behaviour
    productService Reactive & proactive Actor Actor that receives and handles products creation and deletion requests. It associates a PID to the product.
    cargoService Reactive & proactive Actor The mind of the system: it handles requests to cargorobot and consults, updates and manages dbWrapper and cargorobot. It also sends updates to the holdStatusGUI.
    cargorobot Reactive Actor The main entity of the maritime cargo system, which moves within the hold area and performs container delivery to the slot when sonarWrapper signals the arrival of a new container and the sonar detects it.
    sonarWrapper Proactive Actor It wraps the sonar; thanks to it the system can receive a signal when a new container is ready to be loaded.
    alarmDevice Proactive Actor Entity which observes the sonarWrapper and when a distance D > DFREE is measured for more than 3 secs it turns on a led and signals to cargoService the sonar failure and the consequent halt to all its activities. As soon as D ≤ DFREE it turns off the LED and signals to cargoService to resume the system activities.
    holdManager Reactive Actor Entity that manages the state of the hold. It reacts to requests from cargoService answering whether there is a free slot or not.
    dbWrapper Reactive Actor It's the interface to the DB, for adding, removing and getting product porpuses.
    holdStatusGUI Reactive Actor Entity that receives data from the cargoService and updates a web-GUI, used to monitor the status of the hold. We model it as an actor beacuse it processes messages one by one and updates the web-GUI accordingly.
    testCargo Proactive (Test)
    Actor    		 Test actor that sends dummy requests to the cargoService in order to verify how the whole system reacts.
    testContainer Proactive (Test)
    Actor    		 Test actor that approaches the ioport and verifies how the system reacts to the arrival of test container.

    3.3 Messages

    Messages Type Sender Receiver Description
    createProduct Request Alien productService Requests a new product entity creation
    createdProduct Reply productService Alien Returns the newly product entity's PID
    deleteProduct Request Alien productService Requests the deletion of a product entity by giving its PID
    deletedProduct Reply productService Alien The product entity identify by PID that has been deleted

    loadRejected Reply cargoService Alien The product can't be loaded on the hold as requested (there isn't any free slot or the weight would be greater than the MAX_WEIGHT)
    distance Dispatch sonarWrapper alarmDevice The distance (in cm) measured by the sonar
    containerWaiting Dispatch sonarWrapper cargorobot A container is waiting at the ioport
    stop Event alarmDevice All listening (cargoservice and cargorobot) Since the sonar could be broken, the system must stop all its operations
    resume Event alarmDevice All listening (cargoservice and cargorobot) Since the sonar is now correctly working, the system can resume its operation

    4. Model

    4.1 Why QAK

    We chose to model the system with the QAK meta-modelling language since we want to follow the microservice and actors paradigms and this DSL (Domain Specific Language) has been developed in order to fill the abstraction gap related to communication between those entities. We also chose QAK for these reasons:
    • It was entirely implemented by our software house, which makes it easier to get information and support from the actual creator of the language.
    • It provides a way to represent concepts related to services, such as request, response, dispatch, etc., while other general-purpose languages would need external libraries (e.g. Java would need some kind of JAR to implement Actors), frameworks or resources to do the same.
    • It allows you to create executable models directly from the language, which capture the salient aspects of the requirements or problems.
    • Our software house already has some project and components we could exploit and use.
    • It's easy to create prototipes to show to the customer

    7. Sprint conclusions

    7.1 Summary

    This first Sprint (Sprint0) grouped the requirements of the project, by outlining the following macro components, ordered by priority:
    1. Core-business (requirements RF1, RF2, RF3, RF4): involves the chain of the main nodes of the system.
      HIGH PRIORITY: the service can't be provided without it.
    2. ioport (requirements RF5, RF6, RF7, RF8): involves the delivery of products to the right reserved slot.
      MEDIUM PRIORITY: the service can receive requests but can't take the product to the reserved slot without it.
    3. GUI (requirement RF10): shows the current state of the hold
      LOW PRIORITY: the system correctly works, this is an "additional service".

    7.2 Open issues

    During the Sprint0 we found out the following open issues to ask to the customer:
    • Alien: which entity gives the input for product creation, deletion and load?
    • Request queue: when a load request arrives when the system is stopped for sensor failure, which policy we should use:
      • The request is processed immediately and rejected. The requester must resubmit the load request.
      • The request is queued and processed once the system becomes available again.
