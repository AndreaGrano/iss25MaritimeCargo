    3. Problem analysis

    3.1 Normal procedure for handling of load requests

    Question: what is the workflow when a loadRequest is submitted by a client?

    The following steps describe the typical sequence of actions that occur when a loadRequest is received for a product to be placed in the ship’s hold:

    1. A loadRequest is issued to the cargoservice for a specific product to be stored in the ship's hold.
    2. The cargoservice verifies with the productservice whether the product has been registered.
      NB: Registration of a product does not imply that a load request must be issued. It is possible that no load request will ever be created for a registered product.
    3. If the product is registered, it requests an available slot to be assigned.
    4. When sonar sensor detects that a new product has arrived at the ioport, i.e. when the sensor measures a distance D < DFREE/2 for a reasonable duration (i.e., 3 seconds), it signals it to cargoservice.
    5. Once detected, cargoservice sends to the cargorobot the slot where to put the container. Starting from its HOME position, cargorobot picks up the product at the ioport, delivers it to the assigned slot, unloads it, and returns to HOME.
    N.B: Any new load request received during the above steps are queued and processed only after the robot has returned to its HOME position.

    3.2 Special cases for load requests

    While the standard procedure ensures smooth product placement in the hold, several exceptional scenarios must be considered to guarantee system robustness, reliability, and safety. The following sections outlines the key exceptions that may occur during the handling of a loadRequest, along with the expected system behavior in each case.

    Problem: what happens when a loadRequest is submitted for an unregistered product?

    If a product is unregistered, the system cannot know its weight and assess whether the load is acceptable in order not to overload the hold. For this reason, in this case a loadRequest must be rejected.

    Problem: what happens when a loadRequest is submitted, but there is no available slot?

    The hold is obviously finite and has only so many spot where products can be loaded (in this case, 4). If a product is eligible for loading, but there isn't any available slot, a loadRequest must be rejected.
    One may propose a system to optimize loading requests in some way (e.g. priority), but it is totally outside of the requirements and nothing will be done in this way.

    Problem: what happens when there is a failure regarding sonar sensor?

    As enforced by the customer, when a failure for sonar sensor is detected (anther the condition in RF11), every functionality of the system must be stopped. In this sense, the system needs to not allow any submission of load requests; therefore, when a failure is detected and until it is solved, any new loadRequest must be rejected.

    3.3 cargorobot behavior during a sonar sensor failure

    Problem: what happens to cargorobot when a sonar sensor failure is detected?

    As enforced by the customer, when a sonar sensor failure is detected the system must stop any of his activities. In this sense, he heavily stressed that even cargorobot must interrupt any of his movement.

    3.4 A small sub-system deployed far away

    The aim of this sprint is to the develop a small sub-system that reads and eventually evaluates distances measured with a sonar sensor. Since it is deployed at the ioport, this node has to be small and efficient. Also, our software house has already developed some usefull python scripts that can be usefull in order interface the system with the physical devices at the ioport (i.e. sonar and LED); those scripts can be found here and are, in particular:

    • sonar.py: a script that read a distance using a sonar connected to a RaspberryPi and writes the data in a text file
    • ledPython25On and ledPython25Off: two scripts that turns on and off, respectively, a LED connected to a RaspberryPi.
    For those reasons, we chose to adopt a RaspberryPi, a small single-board computer (SBC) that is optimal to the ends of this sub-system. It should be noted that the softwares developed by our software house should be compatible with every iteration of this SBC, but they have been largly tested with a Raspberry Pi 3 Model B+.

    3.4.1 Sonar sensor

    In order to interface the system the python script that act as a sort of driver for sonar sensor, abstracting from its physical implementation, we need a new software component that sends the distances that has been read to every other component who is interested in them. For this purpose, we are introducing a new actor, named dbwrapper.

    Question: who does evaluate the distances measured by sonar sensor?

    The distances measured by the sonar sensor are significant for two different scopes:

    • acknowledging the presence of a new product at the ioport (i.e. when a value D < DFREE/2 is measuered for at least 3 secs)
    • evaluating if the sonar sensor has failed (i.e. when a value D > DFREE is measuered for at least 3 secs)
    In order to satisfy those requirements we need two different new software components (one responsability each). We chose to model them as actors, since they have to autonomously interact with other entities of the system in an reactive/active manner (they will receive data from sonarwrapper and send signals according to their roles); they are:
    • containerManager, an Actor that receives data from sonarwrapper, evaluates if a new product container is present at the ioport and eventually signals it to cargoservice
    • alarmManager, an Actor that receives data from sonarwrapper, evaluates if sonar sensor has failed and eventually signals this event to the other entities; also, it signals when the system can resume its acrivities after a previous failure has been solved

    Question: who does turn on and off the LED?

    An immediate choise would be to let alarmManager to turn on and off the LED, according to his evaluation of data. However, maybe it is better to delegate this operation to another software component, either an existing or a newly introduced one, since it seems a very specific behavior (in order to signal a failure) that could be changed in the future. In this sense, alarmManager would remain the same and only the other entity should be changed accordingly.

    4. Project

    We think it's better for this sprint realization to divide implementation into the following parts:

    • Part 1: sonar sensor implementation
    • Part 2: delivering operation

    4.1 Part 1: sonar sensor implementation

    As stated during the problem analysis phase, in order to interface with the data produced by sonar sensor, the system requires a set of specialized software components, each responsible for a distinct aspect of interaction and control.
    These entities collectively ensure correct detection of container presence at the ioport and sonar sensor fault monitoring.
    So, we implemented the following Actors:

    • sonarWrapper: this is an actor which reads distances from a text file produced by the script sonar.py, in order to remain hardware-agnostic. It performs periodic sampling of the measured distance, once every second, and emits distance update events
    • containerManager: this is an actor that collects distance events emitted by sonarWrapper and evaluates whether a product container is present at the ioport (i.e. when a distance D < DFREE/2 has been measuered for at least 3 secs). If so, it sends a containerwaiting request to cargoservice and waits for the correspondent reply
    • alarmManager: this is an actor that collects distance events emitted by sonarWrapper and evaluates whether the sonar sensor has failed (i.e. when a distance D > DFREE has been measuered for at least 3 secs). If so, it emits a stop interrupt event; when the failure has been solved it also emits a resume event
    • alarmDevice: this is an actor that collects stop and resume events emitted by alarmManager and turn on and off a LED accordingly, using the ledPython25On and ledPython25Off python scripts already developed by our software house

    4.2 Part 2: delivering operation

    In this sprint, the focus shifts to implementing the delivery workflow for the cargorobot, which is responsible for transporting a container from the ioport to an assigned slot.
    The pathfinding and map management logic was already completed in sprint1. The software implementation follows what specified in point 5 of paragraph 3.1.

    4.3 Physical implementation

    For the physical implementation of the sonar we used a HC-SR04 sonar sensor.
    Figure 3 – Sensor HC-SR04

    We connected it to a RaspberryPi GPIO and it is controlled by the means of the Python script sonar.py. We also connected the LED to the Raspberry Pi GPIO.
    Here is the detail of links between Raspberry Pi GPIO and physical components: 
    HC-SR04 sonar's connections:
    • VCC : physical pin 4 (+5V)
    • GND : physical pin 6 (GND)
    • TRIG: physical pin 11 (WPI 0, BCM 17)
    • ECHO: physical pin 13 (WPI 2, BCM 27)

    LED's connections:
    • Anode: physical pin 22 (WPI 6, BCM 25)
    • Catode: physical pin 20 (GND)

    4.4 Messages

    We add the following messages:
    Messages Type Sender Receiver Description
    distance Event sonarWrapper - The distance mesured by the ioport (send each second)
    stop Event alarmDevice - System must stop because of a system failure
    resume Event alarmDevice - System can resume because system failure is solved
    containerwaiting Request containerManager cargoservice There is a container waiting to be loaded at the ioport, cargoservice needs to be informed
    containerloaded Reply cargoservice containerManager The container that was waiting to be loaded at the ioport has been correctly delivered to its assigned slot. After beeing notified by cargorobot, cargoservice tells it to containerManager
    dodelivery Request cargoservice cargorobot cargoservice, informed by containerManager with containerwaiting, orders cargorobot to load the container waiting at the ioport and to deliver it to the assigned slot
    deliverydone Reply cargorobot cargoservice The container has been delivered correctly and the cargorobot has got back to its HOME position

    6. Conclusions

    During this sprint, we successfully completed the implementation of all functionalities related to the ioport management and the container delivery process to the assigned slots. Specifically, the following objectives were achieved:
    • realization of dedicated components to read and analize data produced by the sonar sensor (sonarWrapper, containerManager, and alarmDevice).
    • accurate detection of product presence at the ioport and real-time handling of system failure conditions triggered by sonar sensor anomalies.
    • implementation of the cargorobot logic for performing automated container pickup and delivery, including movement from HOME to ioport, delivery to the assigned slot, unloading, and returning to HOME.
    In addition, we developed and verified all mechanisms for system suspension and resume based on sonar sensor input, ensuring compliance with safety requirements (RF11 and RF12).
    All components were fully tested, both individually and as part of end-to-end system flows, and the sprint goals were completed within the planned timeframe.
    This sprint represents a significant milestone toward a fully autonomous and sensor-driven cargo handling system.
